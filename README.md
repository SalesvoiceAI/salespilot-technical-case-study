# SalesPilot Technical Case Study

> 採用・技術評価目的の公開ケーススタディです。ソースコードは非公開です。

---

## 概要

複数の不動産会社を単一基盤で運用する**マルチテナント型 AI アウトバウンド架電自動化 SaaS**。

顧客リストをもとに AI ボットが自動発信・会話・意向分類を実行し、商談温度の高いリードを通話中にリアルタイムで担当営業へ通知する。担当者は「興味あり」と判定された案件のみに集中できる体制を実現する。

## 解決している課題

| 課題 | 従来の状態 | 本システムによる改善 |
|---|---|---|
| 架電件数のばらつき | 担当者のスキル・意欲に依存 | リストベースで定量的に消化 |
| リード温度感の見落とし | 担当者の主観・記録漏れ | LLM + ベクトル検索による即時タグ付け |
| 通知の遅延 | 日報・会議での共有 | 通話中にプッシュ通知 |
| 複数社への対応 | 会社ごとに個別構築が必要 | 設定変更のみで新テナント追加 |
| 時間外架電の制御 | 手動対応 | 会社別スケジュールによる自動制御 |

## ソリューション

顧客への最初のコンタクトを AI に委譲し、人間の担当者は温度感の高いリードだけを受け取る設計を採用した。

**3つの核となるアプローチ:**

1. **構造化された会話フロー**: ステートマシンで会話の進行を管理し、顧客の意向に応じてクローズ・転送・再連絡などの分岐を自動制御する
2. **多層インテント分類**: ルールベース → ベクトル検索 → LLM の3段階フォールバックで、処理速度と分類精度のトレードオフを最小化する
3. **リアルタイム通知**: 意向ありと判定した瞬間に時限付きリンクを担当者へ即時プッシュする

## アーキテクチャ概要

> アーキテクチャ図は近日公開予定

```
Twilio Cloud (発信・通話管理)
         │
         ▼
  ┌─────────────────────────────────────────┐
  │              Ubuntu VPS                 │
  │                                         │
  │  Nginx (SSL終端) ──► TwiML サーバー      │
  │                           │             │
  │                    Twilio 音声ストリーム  │
  │                           │             │
  │                    WebSocket サーバー    │
  │                           │             │
  │          ┌────────────────┴──────────┐  │
  │          │  Deepgram  (STT)          │  │
  │          │  Azure OpenAI (LLM)       │  │
  │          │  Pinecone (ベクトル検索)  │  │
  │          │  ElevenLabs (TTS)         │  │
  │          └──────────────┬────────────┘  │
  │                         │               │
  │   インメモリ状態管理 ◄──┤               │
  │   永続化ログ DB    ◄────┘               │
  └─────────────────────────────────────────┘
                    │
                    ▼
         LINE Messaging API → 担当営業
```

**主な設計方針:**
- **同期的なリアルタイム処理**: STT → 分類 → 応答生成 → 音声再生を1通話セッション内で完結させる
- **非同期ログ記録**: クリティカルパスからログ書き込みを分離し、通話品質への影響を排除する
- **マルチテナント分離**: 音声プロファイル・架電スケジュール・API 割り当てを会社単位で独立管理する

## 技術選定理由

### Deepgram（STT）
日本語の電話音声をリアルタイムストリーミングで処理できる STT として採用。ローカルモデルと比較して遅延・精度のバランスが高く、ストリーミング API が会話フロー制御に必要な発話終端の確実な検出をサポートしている。

### Pinecone + Azure OpenAI Embedding（一次分類）
顧客の同意・拒否表現は表記揺れが大きくルールベースでは網羅が困難なため、意味的類似度検索で高カバレッジを実現。Embedding モデルを LLM と同一プロバイダに統一し、API 管理コストを削減した。

### 正規表現ルール（最優先層）
確実性の高い意図（日付表現・明確な同意語など）は、LLM を経由することで逆に遅延とハルシネーションリスクが増す。ルールベースを最上位に置くことで速度と確実性を両立し、LLM の呼び出し回数を最小化した。

### ElevenLabs（TTS）
会社ごとに音声プロファイルを分離できる点が採用の決め手。通話中の動的フレーズ生成に対応し、事前収録音声と組み合わせた2段階の音声提供が可能。低レイテンシモデルを選択し、自然な会話テンポを維持した。

### Redis（インメモリ状態管理）
通話セッション中に高頻度でアクセスする顧客・会社情報を TTL 付きでキャッシュし、DB へのラウンドトリップを排除。永続化ストレージとの2層管理によりフェイルセーフを確保した。

## 設計上の工夫

### ステートマシン型会話制御
会話フローをステップ × 分類カテゴリの遷移テーブルとして外部定義し、ビジネスロジックと処理エンジンを分離した。これにより、営業スクリプトの変更がコード修正を必要としない構造になっている。

### 3層インテント分類
ルールベース → ベクトル検索 → LLM の順に評価し、前段でヒットした場合は後段を呼ばない設計。速度重視の処理から精度重視の処理へ段階的にフォールバックすることで、全体のレイテンシを最小化しつつ分類精度を確保する。

### 転送フローの2段階設計
音声再生と電話転送は非同期的に発生するため、ステータスコールバックをまたいだ状態管理が必要になる。フレーズ再生済みフラグを通話セッションに紐付けて管理し、コールバック受信時に転送を発火させる設計とした。

### 非同期ログ記録
STT → 分類 → 応答生成のクリティカルパスに DB 書き込みを挿入せず、専用スレッドで非同期実行する。これにより通話品質に影響を与えることなくフル監査ログを保持できる。

### マルチテナント API 管理
複数社が同一 LLM インフラを共有する際のアクセス制御を、会社単位のデプロイ割り当て方式で実現。設定変更のみで新テナントの追加や割り当て調整が可能な構造にした。

## 技術的課題と解決

### LLM ハルシネーションによる誤分類
**課題**: LLM が定義外の分類文字列を返し、ステップ遷移が壊れる  
**解決**: 許可リストとの照合を挟み、不一致はシステム定義の安全値に置換。API のコンテンツフィルタによる空応答も同様に処理する防御的設計を採用した。

### 同一応答の無限ループ
**課題**: 顧客が同一カテゴリの応答を繰り返し、会話が終了しない  
**解決**: カテゴリ別の発言回数をセッション内でカウントし、閾値超過で強制クローズに遷移するフロー制御を実装した。

### 音声ストリームとコールセッションの紐付けタイミング
**課題**: 音声ストリームの識別子と通話の識別子が別系統で発行され、マッピング確立前に音声データが届く  
**解決**: ストリーム開始イベントを受信したタイミングで双方向マッピングを確立し、それ以前のデータは安全に棄却する。また TwiML の構造を修正し、通話 ID をストリーム開始イベントに確実に含める設計とした。

### 転送失敗時の UX
**課題**: 転送先不在の場合、顧客に何も通知されないまま通話が切れる  
**解決**: 転送完了コールバックで結果を判定し、失敗時は専用フレーズの再生と別クローズフローへの誘導を2段階で実行するフォールバックを設計した。

### 日付表現の正規化
**課題**: 「GW明け」「連休明け」などの曖昧な表現を LLM が絶対日付に変換できないケースがある  
**解決**: 変換失敗時は元の表現をそのままタグに埋め込み、担当者が判断できる形で通知に含める。完璧な正規化よりも「確実に伝わる通知」を優先する実用的な判断をした。

## セキュリティ・プライバシー配慮

- 顧客情報は時限付きワンタイムリンクでのみ参照可能とし、通知メッセージ本文への直接埋め込みを排除
- 機密ファイルへのアクセスはリバースプロキシおよびアプリケーション層で多重防御
- 全外部 API の認証情報は環境変数で管理し、コードベースへの混入を防止
- 音声データは通話セッション中のみ一時保存し、終了後に廃棄
- 個人情報ログの長期保存を抑制するリテンションポリシーを DB 側で設定
- 各サービスはプロセス管理デーモンで自動再起動し、可用性を確保

## 今後の改善

1. **管理画面の構築**: テナントごとの架電状況・分類結果・通知ログをリアルタイムで可視化するダッシュボード
2. **分類精度の継続改善**: 会話ログをもとにベクトル DB のサンプルを定期更新し、新たな言い回しへの適応を自動化
3. **レイテンシ最適化**: LLM 応答生成のストリーミング対応とキャッシュ戦略の導入
4. **CRM 連携**: タグ・通知データを既存の営業管理ツールへエクスポートする API の提供
5. **多言語対応**: STT・TTS・LLM プロバイダの切り替えを設定ベースで抽象化し、日本語以外の市場への展開を容易にする
